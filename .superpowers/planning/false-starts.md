# False Starts

## 2026-07-06 - In-memory queue + Realtime listener + circular-import fix for WhatsApp notifications

- Type: False Start
- Scope: IT-Ticket project — `bot/` WhatsApp notification process (Baileys)
- Confidence: High
- Evidence: `docs/superpowers/specs/2026-07-06-wa-notification-bot-design.md` and `docs/superpowers/plans/2026-07-06-wa-notification-bot-plan.md` (both now marked Superseded). That plan diagnosed "sendMessage succeeds but message not delivered" and "circular import between index.ts and supabaseListener.ts causing a stale `waSocket` binding," and prescribed: keep an in-memory `messageQueue`, fix the stale binding via a setter/getter module (`bot/notifier.ts`, never actually created — the getter fix landed directly in `bot/supabaseListener.ts` instead), and tune Baileys connection options. This was partially executed (uncommitted working-tree changes to `bot/index.ts`/`bot/supabaseListener.ts` at the time) and the user still reported notifications "tidak bekerja sama sekali" — the whole approach was declared a failure and replaced by a durable-outbox architecture (`docs/superpowers/specs/2026-07-06-whatsapp-notification-rebuild-design.md`).
- Rule for future plans: for this project, do not re-propose an in-memory/process-local queue as the source of truth for WhatsApp notification delivery, and do not rely solely on Supabase Realtime `postgres_changes` subscriptions (no error-handling on `.subscribe()`) as the trigger mechanism for anything business-critical. The durable-outbox + polling-dispatcher pattern (Postgres trigger enqueues, bot polls, RPC-based atomic claim) is the current accepted architecture — treat any future "bot won't send" report by checking `notification_outbox`/`bot_heartbeat` state first, not by re-adding another queue-shaped patch on top of the old design.
- Expiry: Re-check after the rebuild (`2026-07-06-whatsapp-notification-rebuild-plan.md`) has been executed and verified end-to-end in production; if the new architecture also fails, this entry should be revisited rather than assumed still correct.

## 2026-07-08 - Durable-outbox rebuild fixed delivery reliability but not WhatsApp account safety — different problem, needed separately

- Type: False Start (partial — the rebuild itself was not wrong, but treating "notifications now work reliably" as "done" was premature)
- Scope: IT-Ticket project — `bot/` WhatsApp notification process
- Confidence: High
- Evidence: the durable-outbox rebuild (see the other 2026-07-06 entry in this file) was live-tested for the first time and DID work end-to-end (trigger → outbox → dispatcher → WhatsApp send, confirmed via bot logs showing "Sent outbox message..."). But within seconds of the fresh QR pairing, the bot's WhatsApp number got suspended by WhatsApp itself — a burst of near-identical messages to multiple individual admin numbers with a fixed 1.5s delay, right after a fresh device pairing, is a documented WhatsApp bot-detection pattern (confirmed via 2026 community sources: fixed/mechanical delay intervals, bulk-send-immediately-after-pairing, and near-identical-content-to-many-recipients are all independently flagged signals). The rebuild solved "does the message get delivered reliably" but never addressed "does sending this way get the account banned" — a completely separate risk dimension the original spec's Non-Goals section didn't examine.
- Rule for future plans: when a plan's goal is "make delivery reliable" for a proactive/outbound messaging bot on an unofficial/reverse-engineered protocol (Baileys, whatsapp-web.js, WAHA, etc.), reliability and platform-ban-avoidance are two distinct requirements that must BOTH be in scope from the start — a spec that only covers durability (retry, outbox, crash recovery) and is silent on send-pattern/velocity/fresh-session behavior will pass every review checklist and still get the account banned on first real use. For this project specifically: admin notifications now route to one WhatsApp group instead of broadcasting to N individual numbers (`notification_settings.admin_group_jid`, migration `007_admin_group_notifications.sql`), per-message delay is randomized 3-5 minutes (not fixed), and a 2-hour warm-up gate suppresses all sending immediately after a fresh QR pairing (`isNewLogin` in `bot/connection.ts`). See `docs/RISK_REGISTER.md` ("Nomor WA Diblokir") and `docs/DECISION_LOG.md` (2026-07-08 second entry) for the full incident record.
- Expiry: Re-check after this project has run the group-based, warm-up-gated approach in real production for a few weeks without further suspensions. If suspensions recur despite these mitigations, the next escalation per community research is: this is an inherent risk ceiling of any unofficial WhatsApp protocol library for proactive (bot-initiated) messaging, and moving to the official WhatsApp Business Cloud API becomes the recommended fix rather than further pacing tweaks.
